# PART B – REPOSITORY & SYSTEM STABILIZATION (Phase 41)

## B1. ROUTE STABILITY FIX (MOST CRITICAL)

- **Create a single route registry** (a TypeScript object or JSON file) that lists every frontend route and its corresponding backend endpoint(s).  
- At startup (backend), verify that all registered backend routes are actually mounted. If a route is missing, log a clear warning and create a placeholder that returns a standard JSON error with `success: false`.  
- Every frontend navigation button/link must check against this registry. If a route doesn’t exist, the button must be **disabled** with a tooltip, never lead to a blank page.  
- Enhance the 404 handler: detect the user’s role, suggest a relevant safe page, and provide a search bar. Never show a generic “Not Found” without guidance.

## B2. ROLE ENGINE FIX (ELIMINATE UNDEFINED ROLE CRASHES)

- Build a **central role resolver** (`resolveRole(user)` → `"admin" | "teacher" | "assistant" | "student" | "parent" | "guest"`).  
- **Rules:**  
  - Never trust the frontend role alone. Always validate the role from the backend (JWT token or session).  
  - If the role is undefined or invalid → treat as `"guest"` and redirect to login with a clear message.  
  - Log any invalid role attempts to the audit log.  
- Apply the resolver in a global React context and in a backend middleware. Every protected component/page must use this resolved role, not a raw `user.role` property.

## B3. 404 ELIMINATION SYSTEM

- Implement a startup **route‑checker** that scans all backend route registrations and compares them with the frontend API call map. Output a report to the admin debug panel.  
- For any unregistered frontend call, automatically create a fallback API endpoint that returns a standard error `{ success: false, error: "endpoint not yet available" }` and log the missing route.  
- In production, replace all broken frontend links with either a working route or a “Coming Soon” / “Feature Unavailable” fallback UI.

## B4. DEPLOYMENT COMPATIBILITY FIX

- Ensure the application can be deployed on **Replit, Railway, Render, or a basic VPS** without code changes.  
- Use a single `PORT` environment variable. In production, the backend serves the built frontend as static files. In development, the Vite dev server proxies API requests.  
- Validate all required environment variables at startup (`DATABASE_URL`, `JWT_SECRET`, `OPENAI_API_KEY`, `APP_URL`, etc.) and exit with a clear error message if any are missing.  
- Add a `/health` endpoint that returns `{ status: "ok", db: "connected", uptime }`.

## B5. UI ROUTE SAFETY LAYER

- Every button and link in the application must be wrapped with a safety check: `if (!routeExists(target)) { showFallbackUI() }`.  
- Build a small utility `isRouteValid(path)` that consults the route registry.  
- Apply this utility to the sidebar, all dashboards, and any navigation components.

## B6. DATA INTEGRITY FIX

- Run a database audit to ensure:  
  - Every student belongs to a valid account.  
  - Every enrollment links a valid student and a valid course.  
  - Every course belongs to a valid teacher.  
  - Every session slot links to a valid course and teacher.  
  - Every attendance record links to a valid session and student.  
- Provide an admin “Repair Orphans” button that removes or fixes broken relationships (with confirmation).  
- Add foreign key constraints / cascading deletes where missing and safe.

## B7. ERROR HANDLING OVERHAUL

- Replace all empty `.catch(() => {})` blocks with proper error logging and fallback responses.  
- Create a global `logError(error, context)` function that stores errors in the `error_logs` table with user ID, route, and timestamp.  
- Wrap the entire frontend in an error boundary that renders the friendly recovery screen and logs the error automatically.

## B8. API CONSISTENCY LOCK

- Standardise **every** API response to:
  ```json
  { "success": true, "data": {}, "error": null }
  { "success": false, "data": null, "error": { "code": "ERR_...", "message": "..." } }
```

· Audit all existing routes and refactor any non‑conforming responses.
· The frontend API utility (fetchJSON) must expect this format and throw a standard error on success: false.

B9. FRONTEND STABILITY FIX

· Apply a universal rule: no component may assume data exists. Every data access must be guarded with if (!data) return <LoadingSkeleton /> or a similar fallback.
· Fix all known cases of Cannot read properties of undefined by adding optional chaining and null‑safe defaults.
· Eliminate loading loops by ensuring TanStack Query’s staleTime and retry are configured globally (e.g., retry: 1, staleTime: 30_000).

B10. PERFORMANCE STABILIZATION

· Remove redundant re‑renders by memoizing heavy components (React.memo, useMemo).
· Reduce animation load by respecting prefers-reduced-motion and disabling heavy animations on low‑end devices.
· Lazy‑load all route‑based components (already partially done with dynamic imports; ensure it’s consistent).

B11. SYSTEM‑WIDE TEST MODE

  · Admin login → dashboard → user management.

  · Student login → enroll → attend → submit → revise.
  · Parent login → pair → view child progress.
· This runner executes real API calls using test accounts (seeded via a script) and logs any failures. It does not replace manual testing but provides a quick health check.

Before completing this phase, verify the following:
