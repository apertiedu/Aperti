## 1. DESIGN SYSTEM — “Liquid Flow”

### Colors
- **Primary (Soft Teal):** `#0D9488` (Tailwind `teal-600`) – use for buttons, active states, highlights, progress bars.  
- **Neutrals:** Black (`#111`), White (`#fff`), Grey `50`, `100`, `200`, `300`, `500`, `700`, `900`.  
- **Semantic:** Success (green), Warning (amber), Danger (red), Info (blue) – only when needed.  
- **No other colors allowed** – no purple, no neon, no custom themes.

### Typography
- **Primary font:** Inter (sans-serif).  
- Weights: Light (300), Regular (400), Medium (500), Semibold (600), Bold (700).  
- Hierarchy: H1 (2.5rem), H2 (2rem), H3 (1.5rem), H4 (1.25rem), Body (1rem), Caption (0.875rem), Label (0.75rem).

### Component Styles
- **Border radius:** Cards, buttons, inputs – `0.75rem` (rounded-xl). Modals – `1rem` (rounded-2xl).  
- **Shadows:** Very subtle – `shadow-sm` on cards, no large glowing shadows.  
- **Animations:** Framer Motion – duration 150–250ms, ease-out. No bounce, no flashy transitions. Use `AnimatePresence` for page transitions.

### General Rules
- **Calm, professional, fast.** No clutter, no visual overload.  
- **Every page must be responsive** (mobile, tablet, desktop).  
- **Accessibility:** Keyboard navigation, screen‑reader labels, focus indicators.

## 3. AUTHENTICATION & SECURITY

### Login Page (`/login`)
- Centered card on a neutral grey background.
- Fields: Email, Password, “Remember me” checkbox, Forgot Password link.
- Buttons: “Login” (primary teal), “Continue with Google”, “Continue with Microsoft” (latter two can be placeholders with Coming Soon badge if not yet integrated).
- States: loading spinner, error message (invalid credentials), success redirect to dashboard.
- Security: JWT stored in localStorage, sent as `Authorization: Bearer <token>` header. Implement rate limiting on backend (5 attempts per minute per IP). Add suspicious login detection (store IP, user agent) and notify user on new device.

### Registration Page (`/register`)
- Fields: First Name, Last Name, Email, Password (with strength meter), Confirm Password, Role (dropdown: Teacher, Student, Parent), Country, Phone (optional), agree to terms checkbox.
- Live validation with error messages below fields.
- On success, send email verification (mock or real using Nodemailer – if not ready, mark “Coming Soon” but still create account as pending verification).
- After registration, redirect to appropriate onboarding flow.

### Email Verification
- Endpoint `/verify-email?token=...` – marks user as verified and redirects to login.
- If email sending is not implemented, show a notice “Verification email sent (simulated)” and allow manual verification via admin.

## 5. WORKSPACE ARCHITECTURE

- Every user must belong to at least one workspace.  
- When a teacher creates a workspace, they become the owner.  
- Workspaces contain their own roles, permissions, and users.  
- Students can be added to a workspace via invitation or direct enrollment (handled later).  
- Admin can view all workspaces.

---

## 6. ROLE SYSTEM & PERMISSION ENGINE

Define default roles with permissions:

- **Super Admin** – all permissions.  
- **Workspace Owner** – full control within workspace.  
- **Teacher** – create courses, manage students, assignments, attendance, etc.  
- **Assistant Teacher** – limited (manage attendance, view students, but not delete).  
- **Student** – view courses, submit work, view own grades.  
- **Parent** – view linked child’s progress, attendance, reports.  
- **Finance Manager** – billing, invoices.  
- **Support Agent** – helpdesk access.  

Permissions are stored individually (e.g., `create_assignment`, `view_finance`, `manage_users`).  
When assigning a role to a user in a workspace, the admin or owner can override specific permissions.
