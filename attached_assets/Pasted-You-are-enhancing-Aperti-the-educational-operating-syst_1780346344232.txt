## 🔥 PRIORITY 2 – TEACHER COURSE MARKETPLACE

Teachers must be able to create courses, set prices, and have students browse and enroll.

### Backend
- **New table: `courses`** – id, teacher_id, title, description, subject, price_egp, thumbnail_url, created_at, is_published.  
- **New table: `enrollments`** – id, course_id, student_id, status (pending/approved/rejected), requested_at, approved_by (assistant_id nullable).  
- **API endpoints:**  
  - `POST /api/courses` – teacher creates course.  
  - `GET /api/courses` – list published courses (public).  
  - `GET /api/courses/:id` – course detail.  
  - `POST /api/courses/:id/enroll` – student requests enrollment.  
  - `GET /api/teacher/enrollments` – teacher/assistant sees pending requests.  
  - `PUT /api/enrollments/:id` – approve/reject (teacher/assistant).  
- **Permissions:** Only teacher owns their courses. Enrollment requests scoped to teacher’s students.

### Frontend
- **Marketplace Page (`/courses`):** Public list of all published courses with search/filter by subject. Each card shows title, teacher name, price, short description. Click leads to detail page.  
- **Course Detail Page (`/courses/:id`):** Full description, teacher info, price, “Enroll” button. If logged in, student can request enrollment. If not logged in, prompt to login.  
- **Teacher Dashboard – Courses Tab:** Manage own courses (create, edit, delete), see enrollment requests, approve/reject.  
- **Student Dashboard – My Courses:** See enrolled courses and statuses.  
- **Design:** Follow Liquid Flow – cards, teal accents, spring animations on hover.

## 🔥 PRIORITY 4 – STUDENT REGISTRATION & TEACHER APPROVAL

- **Student Registration Flow:**  
  1. Student creates account with email, password, name, and selects a teacher from a list of public teachers.  
  2. After registration, status is “pending”.  
  3. Teacher/assistant sees pending students in **Students** page, can approve/reject.  
  4. Once approved, student can access full features.  
- **Existing student accounts should still work; add a migration to set existing ones as “approved”.**

### API
- `POST /api/auth/student‑register` – creates student account with pending status.  
- `GET /api/students/pending` – teacher sees pending list.  
- `PUT /api/students/:id/approve` – approve student.

## 🔥 PRIORITY 5 – PARENT‑STUDENT LINKING WITH PAIRING CODE

- **Parent Profile:** Parent creates account (role: parent), gets a **unique pairing code** displayed on their dashboard.  
- **Student Side:** In student settings, they can enter the parent’s pairing code. This sends a request to the parent.  
- **Parent Side:** Parent sees pending link requests, can approve/reject. After approval, parent can view that student’s progress in GuardianHub.  
- **Support multiple children:** One parent can link multiple students.

### Backend
- `parent_links` table: id, parent_id, student_id, status, pairing_code, created_at.  
- Generate unique pairing code per parent (e.g., 8‑char alphanumeric).  
- API:  
  - `POST /api/parent/generate‑code` – generate/regenerate code.  
  - `POST /api/student/link‑parent` – student enters code, creates pending link.  
  - `GET /api/parent/pending‑links` – parent sees requests.  
  - `PUT /api/parent/approve‑link/:id` – approve/reject.
